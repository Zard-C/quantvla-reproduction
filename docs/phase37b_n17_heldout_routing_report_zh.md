## Routed tactic analysis

这里额外评估一个从 Phase37A probe set 得到的 task-conditioned routing 规则。注意这里的 routing 不是机器人动作 policy，也不是重新训练模型，而是一个推理实现 tactic 的选择规则：

```text
task0, task8 -> window_5_15
task1, task4, task6 -> speed_only
```

换句话说，敏感任务使用更保守的 window fallback，较稳任务使用 aggressive compile。把对应 case 的结果拼接后得到：

| Deployment point | Success | Avg p50 ms | Speedup vs FP16 p50 | Repairs vs FP16 | Regressions vs FP16 |
| --- | ---: | ---: | ---: | --- | --- |
| FP16 baseline | 13/15 | 110.40 | 1.00 | - | - |
| Speed-only compile | 11/15 | 74.79 | 1.48 | `6:26` | `0:24`, `1:25`, `8:26` |
| Request window 0-20 | 14/15 | 100.98 | 1.09 | `6:26`, `8:24` | `8:25` |
| Routed tactic | 13/15 | 79.07 | 1.40 | `6:26`, `8:24` | `1:25`, `8:25` |

这个结果把三种部署模式区分得比较清楚：

- `speed_only` 是 speed-first 模式，p50 latency 最低，但 closed-loop regression 最多。
- `window_0_20` 是 behavior-first 模式，在 held-out set 上成功率最高，但加速幅度较小。
- `routed tactic` 是 speed-robustness trade-off 模式，保留接近 speed-only 的速度，但不能保证无 regression。

因此，本轮 held-out validation 支持 tactic search 的价值，但也说明从 probe set 选出的 routing 规则不能直接当作 universal tactic。routing 本身也需要 held-out rollout validation。论文中应避免把 routing 表述为最优解，而应表述为一种 closed-loop-aware inference scheduling 机制：根据任务或轨迹敏感性，在多个已验证 tactic 之间选择 Pareto 点。

## 收尾判断

Phase37B 可以作为 N1.7 transfer line 的最终实验。它验证了三个关键判断：

1. 加速扰动的影响不是单调的。`speed_only` 可以修复 `6:26`，但也会造成 `0:24`, `1:25`, `8:26` regression。
2. 保守窗口可以降低闭环风险。`window_0_20` 在 15 个 held-out cases 上达到 14/15，是本轮最稳的单 tactic。
3. task-conditioned routing 有 MoE-like 的设计意味，但路由对象是 inference tactic 而不是模型 expert。它提供高速度折中点，但需要 held-out 验证来防止 probe overfitting。

实验层面建议冻结在这里：不再继续扩展 N1.7 tactic sweep，把后续工作转向论文收敛、Pareto 图、算法表述和 limitations。
